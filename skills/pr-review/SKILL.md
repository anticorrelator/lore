# /pr-review Skill

You are running a **holistic multi-lens PR review**. This skill orchestrates the full review pipeline: triage, thematic anchor, adaptive lens selection, parallel lens execution by agent teams, cross-lens synthesis, and structured presentation. It replaces manual lens-by-lens invocation with a single entry point that handles coordination automatically.

For focused single-concern analysis, use individual lens skills directly (`/pr-correctness`, `/pr-security`, etc.). See the integration protocol in `~/.lore/claude-md/review-protocol/lens-workflow.md` for guidance on choosing your entry point.

This skill does not modify source code. Findings are structured and can be posted to GitHub via `post-review.sh`.

## Step 1: Configure

Argument provided: `$ARGUMENTS`

### 1a. Parse PR identifier

Extract the PR number from the first token that matches digits or a GitHub PR URL. If no PR identifier is found, ask the user for the PR number.

Resolve the repo owner/name from the git remote:
```bash
REMOTE_URL=$(git remote get-url origin)
```
Extract `OWNER/REPO` from the remote URL.

### 1b. Detect mode

Parse remaining arguments for mode flags. Exactly one mode applies — if multiple are specified, the highest-priority one wins.

| Flag | Mode | Priority | Effect |
|------|------|----------|--------|
| `--self` | Self-review | 1 (highest) | Adds perspective-lens agents after the parallel lens phase. Self-review blindness mitigation: forces re-examination from alternative viewpoints. |
| `--pair` | Pair review | 2 | Enables turn-based dialog between findings. Each finding is discussed before moving to the next. |
| `--thorough` | Thorough | 3 | Selects all lenses regardless of signal matching. No adaptive selection — every lens runs. |
| (none) | Default | 4 (lowest) | Standard holistic review with adaptive lens selection. |

Store the resolved mode for use in Step 3 (triage) and Step 5 (team spawn).

### 1c. Detect AI authorship

Check for AI involvement using two sources:

**Explicit flag:** If `--ai` is present in the arguments, set `AI_INVOLVED=true`.

**Auto-detection (if no explicit flag):** After fetching PR data in Step 2, scan the PR title and body for keywords indicating AI authorship:
- "generated by", "co-authored with", "AI-assisted", "AI-generated"
- Tool mentions: "copilot", "cursor", "claude", "chatgpt", "codeium", "tabnine"

If auto-detected, present the detection to the user for confirmation:
```
AI authorship detected in PR description: "<matched keyword>"
Activate AI-calibrated review? (hallucination checks, amplified weights, proof evidence) [Y/n]
```

If confirmed (or explicit `--ai` flag), set `AI_INVOLVED=true`. This activates:
- AI-Awareness Calibration from the review protocol (hallucination check, amplified weights, proof evidence requirement)
- Forced lens selection: Correctness and Security always selected regardless of signal matching

### 1d. Record configuration

Store the resolved configuration for use by subsequent steps:

```
PR: #<number>
Repo: <owner>/<repo>
Mode: <default|self|pair|thorough>
AI involved: <yes (--ai flag)|yes (auto-detected: <keyword>)|no>
```

This configuration is carried forward through all subsequent steps. Do not re-parse arguments later — use these resolved values.

## Step 2: Fetch

Fetch all PR data using the shared scripts. Run these in parallel:

```bash
bash ~/.lore/scripts/fetch-pr-data.sh <PR_NUMBER>
```

```bash
gh pr diff <PR_NUMBER>
```

```bash
gh pr view <PR_NUMBER> --json files,title,body,author,commits
```

### 2a. Parse fetched data

From the fetched data, extract:

- **Changed files** — full list with additions/deletions per file
- **PR intent** — title, body, and commit messages describing what the change does
- **Existing reviews** — from `fetch-pr-data.sh` grouped output (`grouped_reviews`, `unmatched_threads`, `orphan_comments`). Filter out outdated threads (`isOutdated: true`). Note any existing review concerns to avoid duplicate findings.
- **Diff stats** — total LOC changed (additions + deletions) for triage in Step 3

### 2b. Complete AI auto-detection

If `AI_INVOLVED` was not set by an explicit `--ai` flag in Step 1, scan the PR title and body now for AI authorship keywords (see Step 1c). If detected, prompt the user for confirmation before proceeding.

### 2c. Diff delivery for lens agents

The full diff must be available to lens agents spawned in Step 5. Strategy depends on diff size:

- **Standard diff (<=400 LOC):** Pass the diff inline in the lens agent task description. This avoids file I/O and keeps agent prompts self-contained.
- **Large diff (>400 LOC):** Write the diff to a temporary file and pass the path to lens agents:
  ```bash
  gh pr diff <PR_NUMBER> > /tmp/pr-review-<PR_NUMBER>.diff
  ```
  Each lens agent reads the file path instead of receiving the full diff in its prompt. This prevents context inflation in agent task descriptions.

Store the diff delivery method (inline or file path) for use in Step 5.

## Step 3: Triage

Apply the Risk-Tier Triage protocol from `~/.lore/claude-md/review-protocol/risk-triage.md` to classify the PR and select lenses. This step is a hard gate — no lens work begins until the user confirms the triage.

### 3a. Classify risk tier

Using the fetched data from Step 2, determine:

**Size:** Count total LOC changed (additions + deletions from diff stats).
- 1-200: Standard
- 201-400: Large
- >400: Oversized — flag prominently, recommend splitting

**Change type:** Scan changed file paths and diff content to classify by highest-risk type present:
- **High:** Auth/authz, cryptography, secrets, payment/billing, data migration, security config
- **Standard:** Business logic, API endpoints, data models, infrastructure, CI/CD
- **Low:** Documentation, comments, style/formatting, test-only, patch dependency bumps

**AI involvement:** Use the `AI_INVOLVED` value resolved in Steps 1c/2b.

### 3b. Select lenses

Apply the Adaptive Lens Selection criteria table from the review protocol. This is a lookup, not a reasoning task.

**If mode is `--thorough`:** Select all lenses. Skip signal matching.

**Otherwise:** Start with the default set (Correctness + Regressions + Test Quality + Interface Clarity), then:
1. For each remaining lens (Security, Blast Radius), check trigger signals against the PR's changed files and diff content
2. If `AI_INVOLVED=true`: force-add Correctness and Security regardless of signals
3. If risk tier is High: force-add Security regardless of signals
4. Apply skip conditions — only skip a lens if ALL its skip conditions are true

### 3c. Present triage to user

Display the triage summary and wait for confirmation:

```
## Triage: PR #<number> — <title>

Risk tier: [High/Standard/Low]
Size: [N LOC] — [Standard/Large/Oversized]
AI involvement: [Yes (--ai flag) / Yes (auto-detected: <keyword>) / No]
Change types detected: [list]

### Selected lenses

| Lens | Reason |
|------|--------|
| Correctness | Default selection |
| Interface Clarity | Default selection |
| Security | AI involvement flag + auth changes detected |
| Regressions | Default selection |
| Test Quality | Default selection |

Proceed with this lens set? You can add or remove lenses before we begin.
```

**This is a gate.** Wait for the user to confirm, add lenses, or remove lenses before proceeding to Step 4. If the user adjusts the lens set, update the selection accordingly — do not re-run the criteria table.

If the diff is >400 LOC, include a note:
```
Note: This PR exceeds 400 LOC. Defect detection rate decreases significantly at this size.
Consider splitting the PR if feasible. Proceeding with full review.
```

## Step 4: Thematic Anchor

Run the thematic analysis inline (not as a spawned agent). This always runs, regardless of whether the thematic lens was selected in Step 3. The thematic pass serves as the anchor that orients all subsequent lens agents — its output is injected into every lens agent's prompt.

### 4a. Theme synthesis

Read the PR description and commit messages. Synthesize the stated theme/goal as a single clear statement: "This PR [verb] [what] [why]." If the PR description is vague or missing, infer the theme from the commit messages and changed files, but note the gap.

### 4b. Per-file alignment map

Walk each changed file and classify its relationship to the theme:

- **Directly supports** — necessary to achieve the stated theme; without it, the theme is incomplete
- **Tangentially related** — related to the theme's area but not strictly required (opportunistic cleanup, adjacent improvements)
- **Unrelated** — no connection to the stated theme (formatting in untouched areas, orthogonal features)

For large diffs (>15 files), group by directory/module first, then classify groups. Apply per-file analysis only to tangential or unrelated groups.

### 4c. Scope verdict

Based on the alignment map, produce one of:
- **Coherent** — all or nearly all changes directly support the theme
- **Mixed** — some tangential changes present but the core theme is clear
- **Scattered** — significant unrelated changes; theme is diluted or unclear

### 4d. Design signals

Identify signals from the thematic analysis that downstream lens agents should know:
- Architectural patterns or conventions observed in the change set
- Cross-cutting concerns (e.g., "this PR modifies both the CLI and the underlying library")
- Areas of higher risk or complexity based on alignment classification
- Missing pieces that other lenses should verify (e.g., "new public API added but no tests observed")

### 4e. Write thematic findings

Build thematic findings JSON conforming to the Findings Output Format in `~/.lore/claude-md/review-protocol/findings-format.md`:

```json
{
  "lens": "thematic",
  "pr": <PR_NUMBER>,
  "repo": "<OWNER>/<REPO>",
  "findings": [...]
}
```

Typical findings: scope creep (suggestion to split), missing pieces that leave the theme broken (blocking), unclear PR description (question).

Create or update the shared work item:
```
/work create pr-lens-review-<PR_NUMBER>
```

If it already exists, load it. Append findings under a `## Thematic Lens` heading in `notes.md`.

### 4f. Prepare thematic context for lens agents

Structure the thematic output as a reusable block that will be injected into each lens agent's prompt in Step 5:

```
## Thematic Context

**Theme:** <theme statement>
**Scope verdict:** <coherent|mixed|scattered>

**Alignment map:**
| File | Classification | Notes |
|------|---------------|-------|
| path/to/file.ext | Directly supports | <brief rationale> |
| ... | ... | ... |

**Design signals:**
- <signal 1>
- <signal 2>
```

This block gives each lens agent the full thematic picture without requiring them to re-analyze scope. They can focus entirely on their specific analytical concern.

## Step 5: Team Spawn

Spawn parallel lens agents to execute the selected lenses from Step 3. Each agent receives everything it needs in its prompt — no references to external files that agents would need to read.

### 5a. Read lens methodologies

For each selected lens, read its Step 3 methodology from the corresponding SKILL.md:

| Lens | Source | Step 3 heading |
|------|--------|---------------|
| Correctness | `skills/pr-correctness/SKILL.md` | Correctness Analysis |
| Interface Clarity | `skills/pr-interface-clarity/SKILL.md` | Interface Clarity Analysis |
| Security | `~/.lore/claude-md/review-protocol/security-methodology.md` | Security Lens Methodology |
| Blast Radius | `skills/pr-blast-radius/SKILL.md` | Blast Radius Analysis |
| Regressions | `skills/pr-regressions/SKILL.md` | Regressions Analysis |
| Test Quality | `skills/pr-test-quality/SKILL.md` | Test Quality Analysis |

Read each selected lens's Step 3 content. You will embed this verbatim in the agent task description.

### 5b. Assemble lens agent prompts

For each selected lens, create a task with this structure:

```
# <Lens Name> Lens — PR #<number>

You are a lens review agent analyzing PR #<number> in <owner>/<repo>.
Your sole focus is the <lens name> lens. Apply only this methodology.

## PR Context
- **Title:** <title>
- **Author:** @<author>
- **Files changed:** <count>
- **Existing review concerns:** <summary of relevant prior comments, or "None">
- **AI involvement:** <yes/no — if yes, apply hallucination check per AI-Awareness Calibration>

<thematic context block from Step 4f>

## Diff

<inline diff for <=400 LOC, or:>
Read the diff from: /tmp/pr-review-<PR_NUMBER>.diff

## Methodology

<verbatim Step 3 content from the lens's SKILL.md>

## Output

Produce findings JSON conforming to the Findings Output Format:
- lens: "<lens-id>"
- pr: <number>
- repo: "<owner>/<repo>"
- Severity: blocking / suggestion / question (default to suggestion when uncertain)
- Each finding: severity, title, file, line, body, knowledge_context

Every finding with severity `blocking` or `suggestion` MUST include a `**Grounding:**` line in the body stating the concrete basis for the severity claim:
- blocking: `**Grounding:** <what breaks> for <whom> when <conditions>.`
- suggestion: `**Grounding:** <specific improvement> benefits <beneficiary>.`

Findings without a `**Grounding:**` line will be downgraded or dropped during synthesis.

Query the knowledge store for each finding:
```bash
lore search "<topic>" --type knowledge --json --limit 3
```

Write your findings to the work item under `## <Lens Name> Lens`:
```
/work update pr-lens-review-<PR_NUMBER>
```

Report back with your findings JSON when complete.
```

### 5c. Create team and spawn agents

Create a team for this review:

```
Team: pr-review-<PR_NUMBER>
```

Create one task per selected lens using the assembled prompts from 5b. Spawn lens agents in parallel. Maximum 5 concurrent agents — if more than 5 lenses are selected (unlikely but possible with `--thorough` + `--self`), batch into rounds.

### 5d. Self-review perspective lenses (--self mode only)

If mode is `--self`, after the standard lens agents are spawned, also spawn perspective-lens agents. These apply the standard lens findings but from alternative viewpoints to counter self-review blindness:

**External reviewer perspective:** "Review these findings as if you are seeing this code for the first time, with no knowledge of the author's intent beyond what the PR description states. Flag any finding where the explanation relies on context not available in the diff."

**Weakest assumption probe:** "For each finding classified as suggestion, ask: what is the weakest assumption this finding depends on? If that assumption is wrong, does the severity change to blocking?"

**Cross-boundary invariant trace:** "For each file in the diff, identify what external code depends on it. Trace one level of callers/consumers. Flag any dependency where the change could alter behavior without the dependent code being updated."

Perspective-lens agents receive the thematic context block and the standard lens findings (once available). They produce additional findings using the same JSON format with `lens` set to `perspective-external`, `perspective-assumption`, or `perspective-boundary`.

## Step 6: Collect Findings

Monitor lens agents as they complete. Collect findings and write them to the shared work item.

### 6a. Track outstanding lenses

Maintain a checklist of spawned lens agents and their status:

```
Lens agents:
- [x] Correctness — completed (3 findings)
- [ ] Security — in progress
- [x] Regressions — completed (0 findings)
- [ ] Test Quality — in progress
```

Update this as agents report back. If an agent fails or times out, note the failure and proceed with available findings — do not block the entire review on one lens.

### 6b. Append findings to work item

As each lens agent reports its findings JSON, verify it conforms to the Findings Output Format (has `lens`, `pr`, `repo`, `findings` fields). Then append to the work item under the appropriate heading:

```
/work update pr-lens-review-<PR_NUMBER>
```

Each lens gets its own heading (`## Correctness Lens`, `## Security Lens`, etc.) with the findings JSON in a fenced code block.

### 6c. Spawn perspective lenses (--self mode)

If mode is `--self` and all standard lens agents have reported, now spawn the perspective-lens agents defined in Step 5d. Pass them the collected standard lens findings as input context. Wait for them to complete and append their findings to the work item under `## Perspective Lenses`.

### 6d. Clean up team

When all agents (standard lenses + perspective lenses if applicable) have reported:

1. Send shutdown requests to all lens agents
2. Delete the team: `pr-review-<PR_NUMBER>`
3. Clean up the temp diff file if one was created:
   ```bash
   rm -f /tmp/pr-review-<PR_NUMBER>.diff
   ```

Proceed to Step 7 with the full set of findings from all lenses.

## Step 7: Synthesis

Read all lens findings from the work item and apply the Cross-Lens Synthesis rules from `~/.lore/claude-md/review-protocol/cross-lens-synthesis.md`. This step transforms independent lens outputs into a unified review.

### 7a. Load all findings

Read the work item to collect all lens findings:

```
/work show pr-lens-review-<PR_NUMBER>
```

Parse the findings JSON from each lens heading. Build a combined list of all findings across all lenses.

### 7b. Identify compound findings

Apply the compound finding rules from the Cross-Lens Synthesis protocol:

1. Group findings by `file`
2. Within each file, identify findings from different lenses whose `line` values are within 3 lines of each other
3. Two or more such findings form a compound finding

For each compound finding, apply the severity elevation table from the Cross-Lens Synthesis section (loaded in Step 2).

Merge compound findings into a single finding with all contributing lens IDs and a merged body preserving each lens's observation.

### 7c. Deduplicate

Apply the deduplication criteria from the protocol: same file, overlapping line (within 3 lines), same severity, and same underlying concern. When deduplicating, keep the more detailed body and add the other lens's ID to attribution.

Do NOT deduplicate findings that address different concerns at the same location — those are compound findings (7b), not duplicates.

### 7d. Produce overall verdict

Summarize the synthesized findings:

```
## Review Verdict: PR #<number>

**Blocking findings:** <count>
**Suggestions:** <count>
**Questions:** <count>
**Compound findings:** <count> (findings flagged by multiple lenses)

**Top concerns:**
1. <highest-severity finding title> — [<contributing lenses>]
2. <second finding> — [<contributing lenses>]
3. <third finding> — [<contributing lenses>]
```

### 7e. Cross-lens theme statement

Synthesize a brief statement about what the lenses collectively found. This is not a repetition of findings — it's a meta-observation about the review:

- Were findings concentrated in specific files or spread across the diff?
- Did multiple lenses converge on the same areas (strong signal) or diverge (broad concerns)?
- Does the finding pattern suggest a systemic issue or isolated problems?

If `AI_INVOLVED=true`, include the proof evidence assessment: which blocking findings have evidence, which are `[unverified]`.

## Step 8: Knowledge Enrichment

Apply targeted knowledge enrichment to the synthesized findings. Individual lens agents already enriched their own findings — this step adds enrichment only where synthesis created new context (compound findings) or where blocking findings need additional citations.

### 8a. Identify enrichment targets

Enrich only these findings:

1. **Compound findings** — newly created by synthesis, not yet enriched by any lens agent
2. **Blocking findings with empty `knowledge_context`** — the lens agent found no relevant knowledge, but the synthesized context may suggest better search terms
3. **Elevated findings** — findings whose severity was raised during synthesis (e.g., suggestion -> blocking)

Skip: suggestions and questions that already have non-empty `knowledge_context` from their originating lens agent.

### 8b. Enrich

For each target finding, query the knowledge store:

```bash
lore search "<finding topic>" --type knowledge --json --limit 3
```

For compound findings, use the merged concern as the search topic (not each individual lens's perspective separately).

Attach relevant citations as `knowledge_context` entries. Follow the output cap from the enrichment protocol: maximum 3 entries per finding.

### 8c. Staleness check

If any knowledge entry is marked STALE and the PR contradicts it, flag as "convention may need updating" — not "PR is wrong." Update the finding body to note the staleness.

### 8d. Update work item

Write the enriched synthesized findings back to the work item under a `## Synthesized Findings` heading, replacing the raw per-lens findings with the post-synthesis, post-enrichment result.

## Step 9: Present Findings

Present the synthesized, enriched findings to the user. This is the primary output of the review.

### 9a. Overall verdict header

Lead with the verdict from Step 7d and the cross-lens theme statement from Step 7e:

```
## Review: PR #<number> — <title>

**Verdict:** <BLOCKING / CLEAN / SUGGESTIONS ONLY>
**Lenses applied:** <list of lenses that ran>
**Blocking:** <count> | **Suggestions:** <count> | **Questions:** <count>
**Compound findings:** <count>

<cross-lens theme statement>
```

Verdict logic:
- Any blocking findings -> `BLOCKING`
- Only suggestions/questions -> `SUGGESTIONS ONLY`
- No findings -> `CLEAN`

### 9b. Findings by severity

Present findings grouped by severity. Within each group, compound findings appear first (they are stronger signals).

```
### Blocking (<count>)

#### 1. [compound] <title>
**Lenses:** correctness, security
**File:** `path/to/file.ext:42`

<merged body from both lenses>

**Knowledge:** [knowledge: entry-title] — relevance summary

---

#### 2. <title>
**Lens:** correctness
**File:** `path/to/file.ext:87`

<body>

**Knowledge:** [knowledge: entry-title] — relevance summary

---

### Suggestions (<count>)

#### 1. [compound] <title>
...

#### 2. <title>
...

### Questions (<count>)

#### 1. <title>
...
```

### 9c. User interaction

After presenting findings, offer actions:

- **Post to GitHub:** Post findings as a batched PR review via `post-review.sh`. The user can approve, edit, or remove individual findings before posting. Then proceed to Step 12.
- **Create work item:** Proceed to Step 10 for a gated work item, then Step 12.
- **Done:** Skip to Step 12 (Capture Insights).

If posting to GitHub:

```bash
bash ~/.lore/scripts/post-review.sh <findings.json> --pr <PR_NUMBER> [--dry-run]
```

The review state is determined by findings: any blocking -> `REQUEST_CHANGES`, suggestions/questions only -> `COMMENT`, no findings -> `APPROVE`.

## Step 10: Gated Work Item

This step is optional — it runs only if the user chose "Create work item" in Step 9c.

### 10a. Prompt for confirmation

```
Create a work item from these review findings?
This organizes findings into actionable categories (Agreed Changes / Verification Needed / Deferred).
[Y/n]
```

If the user declines, skip to Step 12 (Capture Insights).

### 10b. Create work item

```
/work create pr-review-<PR_NUMBER>
```

Write `notes.md` using the proven pr-revise structure, adapted for review findings:

```markdown
# PR #<NUMBER>: <Title>

> **Review findings.** These findings came from a multi-lens review (diff-level analysis). Items in Verification Needed should be investigated against the full codebase before implementation.

## Goal
Address findings from holistic review of PR #<NUMBER>.
Lenses applied: <list>
Verdict: <BLOCKING / SUGGESTIONS ONLY / CLEAN>

## Agreed Changes
Findings where the fix is clear and verifiable from the diff alone.
- [ ] <finding title> (`file:line`) — <brief description> [<lens>]
- [ ] ...

## Verification Needed
Findings requiring investigation before implementation. Each includes a verification directive.
- [ ] <finding title> (`file:line`) — **Verify:** <what to check> [<lens>] [knowledge: <citation>]
- [ ] ...

## Deferred
Findings that are out of scope or need user input.
- [ ] <finding title> — <reason deferred>
- [ ] ...
```

**Categorization rules:**
- **Agreed Changes:** Blocking findings with clear fixes, suggestions where the right action is obvious. Must be verifiable from the diff alone.
- **Verification Needed:** Compound findings (cross-lens convergence warrants investigation), blocking findings without clear fixes, findings touching cross-boundary invariants. Each gets a `**Verify:**` directive.
- **Deferred:** Questions awaiting author response, suggestions the reviewer disagrees with, findings dependent on design decisions not yet made.

Omit empty sections.

### 10c. Readiness assessment

Default readiness: `spec-needed`. Override to `implement-ready` only when ALL:
- Every finding is in "Agreed Changes" (no Verification Needed items)
- All items are trivially obvious fixes verifiable from the diff alone
- No item touches cross-boundary invariants or shared interfaces

## Step 11: Generate Followup Report

This step is mandatory and must not be skipped.

This step always runs automatically after Step 9c resolves (and after optional Step 10 completes). It produces a structured followup report and persists it via `create-followup.sh`.

### 11a. PR Narrative

Using the thematic anchor output from Steps 4a–4d, synthesize a 2–3 paragraph narrative. Do not perform new analysis — assemble from what was already established:

- **Paragraph 1 — Theme and scope:** Restate the theme statement (Step 4a) and the scope verdict (Step 4c). If the verdict is Mixed or Scattered, note which files are tangential or unrelated and why that matters.
- **Paragraph 2 — Design signals:** Expand on the design signals (Step 4d). Describe the architectural patterns or conventions observed, any cross-cutting concerns, and areas of higher risk or complexity identified in the thematic pass.
- **Paragraph 3 — Alignment highlights (optional):** If the per-file alignment map (Step 4b) contains notable entries — files that are unrelated to the theme, files where the thematic role is surprising, or missing pieces flagged as verification targets — call them out by name. Omit this paragraph if the PR is coherent with no noteworthy alignment observations.

```
## PR Narrative

<2–3 paragraphs as described above>
```

### 11b. Implementation Diagram

Build an ASCII logical flow diagram that shows how the PR's changes work mechanically. This is a behavioral diagram, not a module dependency chart — it should show call chains, state transitions, or data flow depending on what the PR does.

Read diagram conventions:
```bash
cat ~/.lore/claude-md/review-protocol/followup-template.md
```

### 11c. Review Findings

Reproduce all synthesized findings from Steps 7 and 8, grouped by severity. This is a faithful copy of the Step 9b output — no summarization, no omission.

For each finding, include:
- Severity badge: `[BLOCKING]`, `[SUGGESTION]`, or `[QUESTION]`
- Title
- Contributing lens(es)
- File and line reference
- Full finding body
- Knowledge citations (if any)

```
## Review Findings

**Verdict:** <BLOCKING / SUGGESTIONS ONLY / CLEAN>
**Blocking:** <count> | **Suggestions:** <count> | **Questions:** <count>

### Blocking

#### [BLOCKING] <title>
**Lens:** <lens(es)>
**File:** `<file:line>`

<body>

**Knowledge:** <citation — or omit if none>

---

### Suggestions

#### [SUGGESTION] <title>
...

### Questions

#### [QUESTION] <title>
...
```

Omit empty severity sections. For a CLEAN verdict, write: "No findings — PR is ready to approve."

### 11d. Determine severity and suggested actions

Using the synthesized findings from Step 7 and the verdict from Step 9a:

| Review outcome | --severity | --suggested-actions primary type |
|---|---|---|
| All clean / CLEAN verdict | low | approve |
| Suggestions only | medium | comment_on_pr |
| Blocking findings exist | high | create_work_item |
| Deferred items exist (no blocking) | medium | create_work_item |

  --severity <per 11d> \

```

The `--content` value is the report body (without the heading — `create-followup.sh` adds the `# <title>` heading automatically):

```

## Error Handling

- **No gh CLI or not authenticated:** Tell user to run `gh auth login`
- **PR not found:** Confirm the PR number and repo access
- **Empty diff:** PR may have no changes — confirm with user
- **Agent failure:** If a lens agent fails, proceed with available findings and note the gap in the verdict
- **No findings:** Report "Holistic review: no findings across all lenses" — this is a valid outcome

## Resuming

- Load existing findings from the work item
- Offer to run additional lenses or re-run synthesis with new findings
- Append rather than overwrite
