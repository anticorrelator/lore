# Retro Failure Mode Reference

Specific failure patterns discovered across retros. Consult when scoring reveals anomalies — not every pattern applies to every retro.

## A. Backlink Resolution Failures

Affects: Dimension 1 (Delivery), Dimension 3 (Gaps)

**Stale backlinks:** Entry existed once but was moved/renamed by `/renormalize`. The slug was valid at plan-authoring time. Fix: `/renormalize` should update backlinks in active work items. >30% unresolved rate caps Dimension 1 at 2.

**Phantom backlinks:** Plan author cited entries that *never existed* — the slug describes a concept that should have an entry but doesn't. Signals a coverage gap, not infrastructure failure. Affects Dimension 3 more than Dimension 1. Collect phantom slugs as proactive capture candidates.

*Severity modulation for annotation-only delivery:* When delivery mode is annotation-only (no `**Knowledge delivery:** full`) and the spec author wrote inline annotation text for the phantom backlink, the phantom status is substantially lower-severity — the annotation IS the delivery, and workers can't pull the full entry regardless. The phantom only causes a failure if workers explicitly attempt to resolve the backlink and find nothing. Detection: check whether the `## Prior Knowledge` section shows annotation text after the `—` separator. If annotation text is present, score D1 as annotation-only delivery (not as phantom failure). If annotation text is absent (bare `[[backlink]]` with no `—` text), treat as normal phantom. For D3, still flag the missing entry as a proactive capture candidate regardless of annotation presence — the phantom indicates a concept worth formally capturing.

**Misrouted citation (phantom sub-type):** Spec lead wrote a plausible-sounding path that doesn't match actual store structure, but the relevant content EXISTS under a different path. Example: citing `architecture/install-and-deployment.md` when the content is at `architecture/portability-via-lore-scripts-symlink.md`, or `architecture/claude-md-fragment-assembly.md` when it's at `workflows/assemble-claude-md-sh.md`. Distinct from true phantom (no content anywhere) and stale (entry was moved — path was valid at write time). Misrouted citations indicate the spec lead reasoned about what the path *should* be rather than searching the store to find what *exists*. Scoring: same as phantom for D1 (delivery failed), but D3 is lower-severity (content exists, just uncited). Detection: for each phantom, run `lore search <slug-fragments>` — if matching content is found anywhere, it's misrouted, not truly missing. Mitigation: `/spec` synthesis step should verify each backlink with `lore search` before writing the path.

**Detection:** For each unresolved backlink, search by slug fragments across categories. Match found under different path = stale. No match anywhere = phantom. Match found in different category or under different slug = misrouted citation.

**Pipeline delivery failure:** Plan has `**Knowledge context:**` backlinks but task generation produces tasks without resolved knowledge. Silent failure — plan looks complete, tasks look reasonable, workers get zero context. Caps Dimension 1 at 2.

**Partial resolution:** Some tasks in a phase get `## Prior Knowledge`, others don't, despite phase-level backlinks applying to all. Root cause: backlink resolution matches on task keywords rather than inheriting phase-level context. Caps Dimension 1 at 3. Fix: propagate phase-level backlinks to all tasks.

**Annotation-only delivery:** `## Prior Knowledge` sections contain plan-author annotations (one-sentence summaries from `**Knowledge context:**` blocks) rather than full entry content. Format: `**[[knowledge:...]]** — <annotation from plan>`. Workers receive the behavioral directive in compressed form but not supporting reasoning, examples, or caveats. Scoring tiers (use the highest tier whose conditions are all met):
- **Dimension 1 = 5**: Full entry prose inlined. Workers received complete reasoning, examples, and caveats.
- **Dimension 1 = 4**: Annotation-only AND workers demonstrably applied all annotated principles correctly across tasks (check worker Observations for confirmation). Correct application is the evidence that the compressed form was sufficient.
- **Dimension 1 = 3**: Annotation-only AND correct application is partial or unverifiable (e.g., workers made no Observations references, template gap prevented delivery of resolution instructions). Also applies when annotations exist but completeness is low (<80% non-empty phase-specific entries). Default tier for annotation-only when evidence is ambiguous.
- **Dimension 1 = 2**: Annotation-only AND workers misapplied or diverged from annotated principles, OR backlink resolution was unavailable (no worker.md template AND no resolution instruction in ad-hoc worker prompt → annotations were the only delivery with no pull fallback).

**Annotation quality — behavioral vs filing-facing:** Within annotation-only delivery, annotations have two distinct quality tiers. *Behavioral annotations* give implementation directives: "do not call emulator.Write() inside View() — mutations are silently discarded on value receiver." *Filing-facing annotations* describe the entry rather than directing action: "understand why Setsid was added originally before removing it." Filing-facing annotations orient workers to read an entry but don't compress the behavioral content — they convert pull into a two-step lookup rather than delivering the directive inline. This reduces the annotation-only mode's main advantage (orientation without full content loading). Detection: annotations that start with "understand why...", "see how...", or describe the entry topic rather than stating a behavioral directive. Scoring impact: does not independently cap D1 (content was delivered and workers can pull full entry), but prevents reaching D1=4 when Phase 1 and Phase 3 annotations are primarily filing-facing while Phase 2 annotations are behavioral — the weak phases undermine the average.

Distinct from "pointer bypass" (which gives no content at all) and full resolution (which embeds full entry prose). Detection: check if `## Prior Knowledge` section entries contain prose beyond the annotation or just the `[[backlink]] — annotation` format.

**Compound failures:** When multiple backlink failure modes co-occur, the lowest individual dimension cap applies. Do not average — compound failures are worse than their components because each compensating layer (tasks.json pre-resolution → lead prefetch → lead manual embedding) failed in sequence.

**Cross-cutting backlink redundancy:** All phases share the same backlinks with no phase-specific entries. Task descriptions receive identical `## Prior Knowledge` content regardless of phase. Technically correct delivery (0 unresolved, content is relevant), but phases that needed different knowledge got the same content — delivery precision is low. Does NOT cap Dimension 1 (content was delivered and relevant), but note as a spec authoring signal: the plan author didn't identify phase-specific knowledge needs. Flag in D1 narrative when all N tasks share identical backlink sets. Mitigation: during `/spec` synthesis, distinguish cross-cutting references (go in `## Related`) from phase-specific knowledge (go in phase-level `**Knowledge context:**` blocks).

**Exception — differentiated annotation on shared backlink:** When all phases cite the same entry but each phase's `**Knowledge context:**` block extracts a *different* actionable directive from that entry, this is NOT redundancy — it's good spec authoring. A comprehensive architectural entry (e.g., TUI architecture overview) can legitimately serve all phases when each phase's annotation targets the relevant subsection: Phase 1 cites file I/O patterns, Phase 2 cites Update() switch cases, Phase 3 cites the polling batch point, Phase 4 cites the detail model dispatch path. Detection: same `[[knowledge:...]]` backlink appears in all phases, but annotation text after `—` differs across phases. Do not flag as redundancy. This pattern maximizes value from broad entries without requiring the store to be decomposed into per-subsystem entries.

**Exception — single-file prose rewrite cycles:** When all phases edit the same prose file (SKILL.md, protocol document, convention file), cross-cutting backlink redundancy is *expected and correct* — all phases need the same conventions because they're editing the same artifact type. Do not flag as low delivery precision in this scenario. Reserve the redundancy note for multi-file cycles where different phases target different subsystems or artifact types and same knowledge is genuinely an imprecision, not a correct invariant.

**Context backlink contamination:** Related work items (consumer references, linked issues) added to `**Knowledge context:**` blocks instead of `## Related`. These entries have no annotation text — they describe relationships, not design rationale — so they appear as bare backlinks in task `## Prior Knowledge` sections. This inflates the "bare backlink" count and depresses annotation completeness metrics even though the knowledge entries themselves are fine. Detection: bare backlinks in `## Prior Knowledge` that resolve to `_work/` paths (not `conventions/`, `architecture/`, etc.) are context backlinks, not knowledge backlinks. They dilute annotation completeness without delivering actionable content. Scoring impact: subtract context backlinks from the denominator when computing annotation completeness — a bare `[[work:...]]` is a spec authoring error, not a delivery failure of a knowledge entry. Mitigation: during `/spec` synthesis, place consumer/related work item references in `## Related`, never in `**Knowledge context:**`.

**Related-section injection by task generator:** The task generator (`lore work regen-tasks`) injects all `## Related` backlinks from plan.md as bare entries into every task's `## Prior Knowledge` section — even when those entries are proper knowledge-store paths (`architecture/`, `gotchas/`), not `_work/` paths. This produces 5-10 bare backlinks per task that are not actionable phase-specific guidance, only cross-cutting orientation context. Distinct from Context backlink contamination (which is spec-author error) — this is a task generator behavior, not a plan authoring error. Detection: count bare entries in `## Prior Knowledge` that match the plan's `## Related` section; if all bare entries trace to `## Related`, this is generator injection, not author error. Scoring impact: subtract `## Related`-sourced bare entries from the annotation completeness denominator — they are not delivery failures of the spec's `**Knowledge context:**` blocks. When computing annotation completeness, use only entries that appear in per-phase `**Knowledge context:**` as the denominator. Mitigation (spec side): move entries that don't need phase-specific annotation to `## Related` and rely on generator injection for cross-cutting orientation; entries requiring annotation belong in per-phase `**Knowledge context:**` blocks.

## B. Lead Bypass

Affects: Dimension 1 (Delivery), Dimension 5 (Delta)

**Full bypass** (Dimension 1 ≤ 2): Lead's TaskCreate descriptions completely omit `## Prior Knowledge` content. Workers start with zero knowledge context.

**Pointer bypass** (Dimension 1 ≤ 3): Lead includes a reference to pre-resolved knowledge but doesn't embed the actual content. Converts push delivery into pull.

**Justified bypass:** Lead reads target files, accumulates enough context to write code-level instructions (line numbers, exact edits) that are strictly more actionable than abstract knowledge entries. Two sub-levels:
- *Lead-paraphrase* (Dim 1: 3): Worker prompts include guidance traceable to knowledge entries, combined with code-level specifics. Knowledge worked through the lead as intermediary.
- *Full bypass* (Dim 1: ≤ 2): Worker prompts contain only code-level instructions with no traceable knowledge substance.

When justified bypass recurs, the knowledge system's ROI for small/well-specified work items is in plan quality (spec time), not worker efficiency (implementation time).

**Template gap:** When `~/.claude/agents/worker.md` is absent, the lead assembles an ad-hoc worker prompt from scratch. Workers receive structurally variable prompts (no canonical template), which may produce inconsistent `**Observations:**` and `**Changes:**` sections — reducing retro evidence quality. Different from full bypass (the lead still injects knowledge context) but reduces reliability of the reporting structure.

**Template gap → knowledge pull failure (compound):** When the template is absent AND delivery is annotation-only, the canonical worker template's instruction to resolve backlinks (`lore resolve "<backlink>"`) is also missing. Workers have no prompt to pull full entry content — annotation text is effectively the ceiling. This compounds annotation-only delivery: annotation-only normally allows pull as a compensating path; template gap removes that path without the lead knowing. Detection: (a) "File does not exist" for worker.md, (b) tasks.json has annotation-only Prior Knowledge, (c) no worker Observations reference to entry content beyond annotations. Scoring impact: downgrade Dimension 1 by 1 from what annotation-only delivery would otherwise score (e.g., 4→3 if correct application observed; 3→2 if application unverifiable). Mitigation: when worker.md is absent, explicitly include in the ad-hoc worker prompt: "Before starting each task, resolve its `## Prior Knowledge` backlinks with `lore resolve '<backlink>'` to get full context."

Detection: check conversation context for "File does not exist" when reading `~/.claude/agents/worker.md`. Mitigation: create the agent template; until then, note as evidence quality degradation in the retro's evidence summary.

**Template gap + lead-paraphrase compound:** When worker.md is absent, the lead naturally compensates by composing detailed ad-hoc prompts that embed plan context (design decisions, file references, exact instructions). This produces good worker outcomes but completely bypasses the knowledge delivery pipeline — the lead IS the delivery layer. The compound makes it impossible to evaluate whether the knowledge *system* (prefetch, backlink resolution, annotation delivery) would have been sufficient, because the lead's manual composition substitutes for all of it. Detection: worker.md absent + worker prompts contain inline design decisions traceable to plan.md + no `lore prefetch` calls in implementation transcript. Scoring: D1 ≤ 3 (delivery happened but not through the system), D5 inherits spec-completeness ceiling. Mitigation: create worker.md template; when absent, `/implement` lead should still call `lore prefetch` to test the pipeline even if composing prompts manually.

**Lead-paraphrase efficacy signal:** When template gap + lead-paraphrase compound produces zero rework across all tasks (observable in execution-log: all tasks completed with no worker-reported divergences or corrections), this signals that lead-paraphrase may be *sufficient* for well-specified work items where the spec itself is comprehensive. The knowledge system's ROI in this configuration concentrates entirely at spec time (informing design decisions) with negligible marginal value at implementation time. This is NOT a failure to fix — it's an expected ceiling for implementation-time knowledge delivery when spec quality is high. However, it makes the knowledge delivery pipeline's value impossible to measure independently of spec quality. Track across retros: if template gap + lead-paraphrase consistently produces zero-rework outcomes, the worker.md template's value proposition is reporting consistency and retro evidence quality, not implementation quality improvement.

**Detection:** Compare tasks.json descriptions against actual TaskCreate descriptions. Note *justified* vs *accidental*.

**Evidence availability:** TaskCreate descriptions are ephemeral (conversation context only). Lead-bypass detection is only fully verifiable in **same-session retros**. For cross-session retros, infer from: (a) worker observations about missing context, (b) notes.md delivery gaps, (c) absence of knowledge-traceable guidance in worker reports. Cap Dimension 1 at 3 cross-session without direct evidence of delivery.

## C0. Implementation Phase Cold-Start

Affects: Dimension 1 (Delivery), Dimension 3 (Gaps)

**Pattern:** A spec phase targets a file or subsystem that has no knowledge entries — the target is genuinely unmapped territory. The spec correctly delivers whatever global/cross-cutting knowledge exists, but workers receive no orientation specific to the implementation target. Workers discover the target's architecture from scratch, then `/remember` captures it post-implementation. This is distinct from Section C (spec investigation cold-start, where investigators search for domain knowledge before planning) — here the cold-start occurs at implementation time, for a phase target the spec never needed to investigate.

**Detection:** When a phase's `**Knowledge context:**` block contains only cross-cutting entries (global principles, install conventions) and the target file appears in no `related_files` metadata across any existing knowledge entry, that phase is cold-start. Confirm post-hoc: mid-cycle captures with `context` fields referencing the phase's target file for the first time indicate unmapped territory.

**Scoring impact:** Cold-start phases are NOT delivery failures — you cannot deliver knowledge that doesn't exist. Score Dimension 1 based on what was available. Score Dimension 3 based on capture outcomes: if workers produced discoveries that were captured for future cycles, the gap is addressed (D3 = 4); if workers discovered something and it wasn't captured, the gap persists (D3 ≤ 3).

**Cross-cutting redundancy in mixed plans:** When a multi-phase plan has one cold-start phase (no file-specific entries) alongside phases with strong knowledge context, the spec may deliver identical cross-cutting annotations to all phases. The cross-cutting annotation is correct for context-rich phases but filing-facing for the cold-start phase (workers must still read the target file regardless). This is not a spec authoring error — it signals a knowledge coverage gap, not a delivery precision error. Detection: all phases share identical `## Prior Knowledge` content AND one or more phases have no file-specific entries. Note in D1 narrative as "cross-cutting redundancy from cold-start phase, not spec precision failure."

**Mitigation (spec authoring):** When a phase targets a file with no `related_files` hits in the knowledge store, add a note to that phase's task description: "Note: no prior knowledge exists for `<file>` — treat this as unmapped territory and report its structural role, integration points, and constraints in Discoveries." This primes workers without requiring knowledge that doesn't exist.

**Mitigation (task generator):** The `lore work tasks` generator could detect when a phase's target files appear in no knowledge entry's `related_files` and auto-inject the unmapped-territory signal into task descriptions.

## C. Cold-Start and Prefetch Failures

Affects: Dimension 1 (Delivery), Dimension 3 (Gaps)

**Cold-start:** No knowledge exists for the investigated domain. `lore prefetch` returns empty. Valid scenario, but the researcher prompt should note "No prior knowledge found — report patterns for capture." This turns cold-start into a seeding opportunity.

**Prefetch silent failure:** `lore prefetch` exits 0 but returns empty. Lead may silently fall back to manual preambles or skip injection. Caps Dimension 1 at 3 (delivery succeeded manually but pipeline failed). **Disambiguation for spec-only cycles:** When prefetch fails but the spec lead manually retrieved entries and embedded them in tasks.json `## Prior Knowledge` sections, the tasks.json delivery pipeline to future workers is intact — the prefetch failure only affected the spec authoring phase (concordance widening). In this case, apply the cap as: Dimension 1 ≤ 3 for the spec's concordance quality, but note that worker-facing delivery is unaffected. Do not apply the full cap when tasks.json evidence confirms complete delivery to workers. Apply the full cap only when prefetch failure AND no fallback recovery AND tasks.json shows missing or skeletal knowledge sections.

**Prefetch query recall failure:** Knowledge entries exist for the investigated domain, but `lore prefetch` returns empty because the compound multi-word query doesn't match FTS5 index terms. Distinct from cold-start (entries genuinely absent) and silent failure (tool bug). Detection: after prefetch returns empty, check the knowledge index (`_manifest.json` or `_index.md`) for entries in the same domain — if entries exist under different terms/titles, this is recall failure. Scoring impact: recall failure is worse than true cold-start for Dimension 1 because the lead could have tried decomposed single-term queries, browsed the index, or used `lore search` as fallback. When prefetch returns empty, the lead should attempt at least one alternative retrieval before concluding "no knowledge exists." Caps Dimension 1 at 2 when the lead treats recall failure as cold-start without attempting fallback retrieval.

**Implementation-time prefetch recall failure:** Same root cause as above but occurs during `/implement` Step 3.1 when the lead prefetches knowledge for worker prompts. When tasks.json already contains annotation-only `## Prior Knowledge` sections (generated at spec time), the prefetch failure does NOT affect worker delivery — annotations are already embedded. The failure only affects the lead's supplementary `$PRIOR_KNOWLEDGE` injection into the worker template's `{{prior_knowledge}}` slot. Scoring impact: when tasks.json has complete annotations AND prefetch fails at implementation time, do not cap Dimension 1 — the primary delivery path (tasks.json annotations) is intact. The lead's fallback from `lore prefetch` to `lore search` is a friction event, not a delivery failure. Cap Dimension 1 only when tasks.json lacks annotations AND prefetch fails AND no fallback retrieval attempted.

**Spec-cycle cold-start tracking:** Cold-start for >50% of investigations caps Delivery at 3; >75% caps at 2. If the same domain produces cold-starts across multiple specs, create a proactive knowledge-seeding work item.

**Explore agent shell fallback gap:** The researcher template documents `lore search` as a fallback retrieval mechanism when prefetch returns empty. But Explore agents lack Bash access and cannot execute shell commands — the fallback is dead code for non-general-purpose agents. Detection: prefetch returned empty AND researchers were spawned as Explore type. Recovery is impossible after dispatch — the gap must be caught at dispatch time. Scoring impact: caps Dimension 1 at 2 (same as prefetch recall failure with no fallback attempted). Mitigation: when `lore prefetch` returns empty before spawning Explore researchers, the lead must (a) try decomposed single-term queries with `lore search`, (b) browse `_manifest.json` for relevant domain entries, and (c) manually inject any found entries into researcher prompts. Use general-purpose agents only when active runtime knowledge retrieval is needed during investigation (e.g., web research, dynamic search). Do not rely on Explore agents to self-recover from empty prefetch results.

## C2. Spec Investigation Factual Errors

Affects: Dimension 3 (Gaps), Dimension 4 (Alignment)

**Factual propagation:** Spec investigators produce findings that contain factual errors about the codebase (e.g., misclassifying a hook's lifecycle stage, overstating code duplication percentages). These errors propagate through plan.md design decisions into tasks.json and eventually into worker task descriptions. Workers either implement based on wrong assumptions (causing rework) or catch the error during implementation (adding evaluation overhead).

**Detection:** Compare plan assertions against worker observations. When workers report "the plan said X but I found Y", trace the error back to the spec investigation finding. Common patterns: (a) lifecycle/timing misclassification (e.g., SessionStart vs Stop hook), (b) duplication estimates based on conceptual similarity rather than code-level analysis, (c) caller/dependency counts underestimated, (d) greenfield library API name errors (see below), (e) mutation return type annotation errors (see below).

**Mutation return type annotation errors:** Spec investigators assume API symmetry across type families (e.g., "LLM evaluators use `@appendNode(DatasetEvaluatorEdge)`, therefore CODE evaluators should too") without verifying each mutation's actual return type against the schema. Cache update patterns (Relay `@appendNode`, `cache.modify()`) are strictly dependent on the mutation's payload type — if the payload type differs across superficially similar mutations, the annotated cache pattern silently fails. Workers following the annotation attempt the annotated approach, discover it doesn't work, then must investigate the backend to determine the actual return type. Detection: spec annotation contains a cache update directive AND the mutation's return type was not explicitly verified in the investigation findings. Common scenario: "Type A mutation uses cache pattern X → assert TYPE B mutation also uses cache pattern X" without checking whether B's mutation returns the same payload type as A's. Mitigation: before annotating any `@appendNode` or `cache.modify()` pattern in a `**Knowledge context:**` block, run `grep -n "mutation_name" schema.graphql` to confirm the return type. When the return type differs from the analogy source, note the divergence explicitly in the annotation. Scoring impact: same as greenfield API errors — single error caught during implementation with no rework: no D4 cap below 3.

**Greenfield library API errors:** When a spec investigates a library the codebase hasn't used before, investigators may infer API names from documentation, README examples, or analogies to similar libraries — without compiling code to verify. These errors propagate into plan.md as assertions ("initialize with `vt.NewTerminal(w, h)`") and reach workers via task descriptions. The compiler catches them immediately (one-line fix), but every worker in the same phase hits the same correction independently, wasting evaluation cycles. Detection: multiple workers report the same "plan said X, actual API is Y" for the same function. Mitigation: spec investigators should either (a) write a small `go build`-able snippet to verify API names before including them in findings, or (b) mark API-name assertions as "unverified — confirm before implementing" so the first worker's fix propagates to subsequent workers via task description update.

**Scoring impact:** Single factual error caught by evaluation task: Dimension 4 no lower than 3 (plan was mostly correct, error contained). Factual error that caused incorrect implementation requiring rework: Dimension 4 caps at 2. Multiple factual errors: Dimension 4 caps at 1. Greenfield API errors caught by compiler (zero rework): no D4 cap — treat as D3 minor gap.

**Mitigation:** For consolidation/refactoring work items, spec investigations should include concrete overlap measurements (shared function counts, caller grep results) rather than conceptual similarity assessments. Phase 3 "evaluation" tasks serve as a structural safeguard — they catch spec errors before irreversible implementation.

## C3. Spec-Investigation-to-Store Gap

Affects: Dimension 3 (Gaps)

**Pattern:** An insight, gotcha, or convention was discovered and documented in plan.md's `## Investigations` section but was never promoted to the searchable knowledge store. Workers in the same implementation cycle (or future cycles) rediscover the same insight by reading source files directly. Distinct from a true coverage gap (where no one knew the information) — the spec lead explicitly validated the insight but left it in the plan artifact, which is not searched by workers.

**Detection:** When a mid-cycle `/remember` capture is filed, compare each capture's content against the work item's `## Investigations` section. If the investigation findings contain the same factual claim, it's a spec-investigation-to-store gap, not a novel discovery. The `context` field of the capture will reference the work item's own investigation if the worker mentioned it.

**Scoring impact:** Spec-investigation-to-store gaps are genuine coverage failures (Dimension 3) even though no one "missed" the insight — the issue is store promotion, not discovery. However, since the plan.md investigation is available to workers who read it, the gap only causes friction when workers don't read the investigation section before starting. Severity: (a) if workers read the investigation and applied the knowledge correctly → minor gap (Dimension 3 ≤ 4); (b) if workers had to rediscover via file reads → moderate gap (Dimension 3 = 3); (c) if workers hit avoidable bugs due to missing context → significant gap (Dimension 3 ≤ 2).

**Promotion rate:** The expected capture rate for spec investigation findings is 0% unless `/remember` is explicitly invoked after `/spec` completes or at the start of the implementation cycle. Retros should note when >1 spec-investigation-to-store gap is found — this signals a systematic capture shortfall, not isolated oversight.

**Mitigation:** After `/spec` synthesis, run `/remember` scoped to "spec investigation findings — capture non-obvious stable discoveries from ## Investigations sections." Alternatively, add a `/spec` post-step that triggers a capture pass over confirmed investigation assertions. The current `/spec` SKILL.md has a capture step at the end of synthesis but it targets design decisions, not individual investigation assertions.

## D. Delta Modifiers

Affects: Dimension 5 (Delta) interpretation only — does not cap scores.

**Prototype-cascade:** For batched similar tasks, knowledge value concentrates on the first task (the prototype). Subsequent workers reference the prototype as template rather than knowledge entries. Score Delta based on the full batch including prototype, not averaged.

**Same-knowledge saturation:** When all phases operate in the same domain, every phase references the same entries. First worker gets value; subsequent workers see identical content with zero marginal insight. Note in Delta analysis — contribution is front-loaded.

**Meta/self-referential work:** When implementation IS knowledge work (editing knowledge conventions, enhancing the store), Delta will be inherently low (1-2). Workers are domain-native. Score honestly but note classification for trend analysis.

**Meta-work + spec-completeness compound modifier:** When meta-work co-occurs with spec-completeness ceiling, the compound ceiling at D5 = 3 prevails over the meta-work "1-2" guidance — provided delivery quality is clean (D1 ≥ 4, D4 ≥ 4). The meta-work note assumes workers are domain-native AND delivery was minimal; when a spec is high-quality and knowledge delivery is clean, the low Delta reflects spec-completeness, not worker domain familiarity. Detection: both conditions hold AND D1 ≥ 4 AND D4 ≥ 4. Score D5 = 3 via the annotation-only + spec-completeness fast-path; do not lower to 1-2 on meta-work grounds alone. When D1 ≤ 3 or D4 ≤ 3, meta-work's "1-2" remains correct.

**Plumbing-vs-value gap:** When Dimensions 1-4 are 4-5 but Delta is ≤3, delivery mechanics are working but value contribution is low. Consider: (a) was the work item small enough that code alone sufficed? (b) did knowledge contribute at spec time? If so, this is an expected ceiling, not a problem.

**Spec-completeness ceiling:** When a spec's investigation phase is thorough enough to produce a complete implementation recipe (exact line numbers, exact edits, exact function signatures), the knowledge system's value at implementation time is structurally capped — workers follow the recipe rather than relying on knowledge orientation. Delta ≤3 is expected and not a failure. The knowledge system's ROI for this cycle was front-loaded into the spec phase, not the implementation phase. Detection: plan.md contains line-number-level specificity and exact edit instructions for every task; execution-log shows workers made zero divergent choices. This is a sub-case of plumbing-vs-value gap but distinct because it's a sign of *high-quality spec work*, not *low-quality knowledge delivery*. Do not mark as a failure — note in Delta analysis and score Gap/Creation dimension (Dim 3) based on what gaps the implementation phase surfaced for future cycles.

**Annotation-only + spec-completeness compound ceiling (D5 fast-path):** When BOTH annotation-only delivery AND spec-completeness ceiling apply simultaneously, D5 = 3 is the expected and correct score — do not invest retro time hunting for D5 = 4 or D5 = 5 evidence that structurally cannot exist. Detection: (a) no `**Knowledge delivery:** full` in any phase AND (b) plan.md contains exact edit instructions/line numbers for every task. When both conditions hold, score D5 = 3 immediately and note "annotation-only + spec-completeness compound ceiling." The 31-retro empirical baseline confirms D5 has never reached 5; the combination of these two ceilings is the dominant cause. Scoring D5 = 4 in this configuration requires positive evidence that delivery changed a worker's approach (e.g., worker explicitly avoids a wrong path mentioned in an annotation, or Discoveries confirm a specific orientation effect) — absent such evidence, D5 = 3 is the honest score.

**Micro-fix ceiling:** Single-file interactive fixes under ~50 lines done without workers have a structural Delta ceiling of 2, regardless of delivery quality. The knowledge system's ROI on micro-fixes is almost entirely **creation** (new patterns captured) rather than **consumption** (existing patterns applied). Dimensions 1 and 4 are inherently low-signal for micro-fix cycles — score them honestly but note that low scores don't indicate a failure worth fixing. Focus Dimension 3 (gaps) on whether the cycle produced captures that improve future coverage. Detection: work item has 1 phase, 1 task, no workers, notes.md empty, execution-log entries only from `/remember`.

**Sequential same-file phantom work:** When multiple sequential tasks all target the same file, one fast worker typically implements beyond their assigned scope, leaving subsequent workers to claim their tasks and report "no changes needed — already implemented." Observable in retro via worker completion reports containing "no new changes needed." Affects Delta interpretation: idle workers aren't a knowledge system failure, but they indicate over-serialization in task design. Note when >2 workers report phantom completion on the same file. Mitigation at spec time: consolidate sequential same-file edits into a single task, or explicitly note "worker-3 before worker-4 on this file."

**Task generator file_targets parsing bug:** When a task subject contains code expressions (e.g., `change X ?? Y to X || Y`), the task generator may parse the expression fragments as file targets instead of extracting actual file paths from the phase's `**Files:**` field. Observable in tasks.json: `file_targets` contains expression strings rather than file paths. Impact: minimal for implementation (task description is clear regardless), but degrades any tooling that consumes `file_targets` for file-scoping or pre-reads. Detection: `file_targets` entries that don't match `*/*.*` glob pattern. Not a retro scoring issue — note as a task generator quality signal for `/implement` improvement.

**Task generator backlink-syntax contamination:** When a task description contains literal `[[knowledge:...]]` or `[[work:...]]` syntax as documentation examples (e.g., "extract all `[[knowledge:...]]` backlinks using regex"), the task generator's backlink extractor parses these as real backlinks and injects them into `## Prior Knowledge` as bare placeholder entries. Workers see `- **[[knowledge:...]]**` with no annotation — noise that provides no orientation. Distinct from phantom backlinks (which reference real-but-missing entries) and from Related-section injection (which injects valid cross-cutting references). Detection: bare entries in `## Prior Knowledge` whose target is a literal ellipsis (`...`) or a syntax pattern rather than a knowledge path. Scoring impact: does not cap Dimension 1 (placeholder noise doesn't degrade real annotations), but subtract these entries from the annotation completeness denominator — they are parser artifacts, not delivery targets. Mitigation in task generator: skip backlink matches where the target component is `...` or contains only punctuation/ellipsis.

**Task generator empty phases from bullet-list plans:** When plan.md uses plain bullet lists (`- Add X`, `- Update Y`) instead of checkbox syntax (`- [ ] Add X`), `lore work tasks` produces `"phases": []` — zero tasks generated despite well-structured phases with clear items. The `/implement` lead must then create tasks manually via TaskCreate, bypassing the tasks.json → worker pipeline entirely. This means no `## Prior Knowledge` sections are generated, no backlink resolution occurs, and the knowledge delivery path is short-circuited at the source. Detection: tasks.json has `"phases": []` and plan.md has `### Phase N` sections with bullet items. Scoring impact: compounds with template gap — when both tasks.json is empty AND worker.md is absent, the lead is both the task designer and the knowledge delivery layer. Caps Dimension 1 at 3 (same as lead-paraphrase) unless the lead also skips `lore prefetch` (then caps at 2). Mitigation at spec time: use checkbox syntax (`- [ ]`) for all actionable items in plan phases, even for spike/exploration plans. Alternatively, fix the task generator to parse both formats.

**Investigation-originated plan without knowledge context:** When a plan is authored from an extensive investigation (e.g., architecture audit with multiple analysis agents) but the findings are embedded directly in the plan prose rather than referenced via `**Knowledge context:**` backlinks, the task generator produces tasks with "No backlinks found in plan" — even though the plan was deeply informed by knowledge entries at authoring time. The plan effectively absorbs the knowledge and re-emits it as findings, severing the link between store entries and worker delivery. Detection: plan.md has detailed architecture diagrams, concept inventories, and specific findings, but zero `**Knowledge context:**` blocks in any phase. tasks.json shows "No backlinks found in plan" for all tasks despite the plan being knowledge-informed. Scoring impact: Dimension 1 ≤ 2 (same as lead-paraphrase — knowledge reached workers only through the lead's manual composition). Dimension 4 = 3 (knowledge was load-bearing at spec time but citations are implicit, not traceable). Delta inherits spec-completeness ceiling. Distinct from lead-bypass (the lead didn't skip delivery — the plan never specified what to deliver). Mitigation at spec time: audit/investigation plans should include `**Knowledge context:**` blocks in each phase referencing the knowledge entries that informed the phase's findings. Even if the plan restates the findings, the backlinks enable the delivery pipeline to inject supporting rationale and caveats that the plan's summary may omit.

**Execution-log underreporting:** When the `/implement` lead logs only a fraction of task completions to `execution-log.md` (e.g., 1 of 16 tasks), the retro must rely on in-session conversation context for worker observations. This degrades evidence quality for cross-session retros where conversation context is unavailable. Detection: execution-log has fewer `source: implement-lead` entries than completed tasks. Scoring impact: does not directly affect dimension scores (evidence is still available in-session), but note as evidence quality degradation. Mitigation at `/implement` time: Step 4 requires logging every task completion — the lead should write execution-log entries as each worker reports back, not just for the first task. The `/implement` SKILL.md already specifies this; the gap is compliance, not specification.

**Execution-log batch-timestamp collapse:** When a worker completes multiple tasks and reports them in a single batch message (common in heavily serialized graphs where one worker cascades through the dependency chain), the lead's reconciliation step bulk-logs all completed tasks at the same timestamp. The execution-log shows N entries with identical timestamps and no per-task sequencing information. Distinct from underreporting (all tasks ARE logged) — the issue is temporal granularity loss. For cross-session retros, this makes it impossible to reconstruct task completion order or identify which tasks genuinely depended on predecessors vs. were over-serialized. Detection: multiple `source: implement-lead` entries share the same timestamp and cover tasks from different phases. Scoring impact: does not affect dimension scores (content is complete), but degrades evidence for the "shared-artifact sequential bottleneck" and "sequential same-file phantom work" patterns which rely on timing to distinguish intentional sequencing from worker scope creep. Mitigation at `/implement` time: log each task as it arrives from the worker's batch report rather than bulk-logging during reconciliation — even if timestamps are close, separate writes preserve ordering.

**Parallel duplicate implementation:** When two workers both claim and implement the same task simultaneously (race condition in task claiming), the result is duplicate artifacts requiring cleanup — e.g., two generated mutation files for the same operation, conflicting edits to the same component. Observable in retro via execution-log showing two workers reporting completion of the same task subject. Not a knowledge system failure — it's a coordination gap in the `/implement` lead's task assignment. Distinct from "sequential same-file phantom work" (where one worker implements beyond scope): here both workers independently implement the same assigned scope. Impact: wasted worker cycles + manual artifact deduplication by the lead. Mitigation at `/implement` time: when the task graph is heavily serialized (most tasks blocked), avoid spawning more workers than the number of concurrently unblocked tasks. For highly serial graphs (only 1-2 tasks unblocked at a time), spawn 1-2 workers instead of the default 4. Does not affect retro scoring — note as an implementation orchestration signal.

**Auto-approved spec evidence degradation:** When `/spec --yes` or `/spec short --yes` runs, all confirmation gates (strategy, understanding, task review) are skipped. For `/spec short --yes`, the entire cycle is a single-agent read-and-draft pass with zero user interaction. For `/spec --yes` (full flow), researchers still investigate in parallel but the lead auto-proceeds through strategy, understanding confirmation, and task review without user validation. Both modes remove three retro evidence sources: (a) user corrections at the understanding gate (which reveal assumption errors), (b) user strategy input (which reveals design priorities), (c) user task review feedback (which reveals scope misalignment). Additionally, the compressed cycle increases the likelihood that optional steps like `/remember` post-research extraction are skipped. Detection: plan.md has no `## Strategy` section AND notes.md has no session entry from the spec run AND tasks.json was generated in the same session as the plan. For full-flow `--yes`: also check whether investigation findings were collected (they should be — `--yes` skips confirmation gates, not investigation dispatch). Scoring impact: does not directly cap any dimension (the plan artifacts are still evaluable), but note as evidence quality degradation — Dimensions 3 and 4 are harder to score confidently because user validation didn't occur. For D3, the user's confirmation gate is a gap-detection mechanism (they catch wrong assumptions); its absence means gaps may be latent. For D4, alignment is assessed against plan artifacts only, not against user-confirmed understanding — score honestly but note reduced confidence.

**Shared-artifact sequential bottleneck:** When all tasks in a phase write to the same foundational artifact (e.g., a protocol doc, a shared schema file), they must be ordered strictly sequentially regardless of team size — workers 2-N are idle while worker-1 completes the chain. Distinct from "sequential same-file phantom work" (which is about workers running ahead of scope): here the sequencing is intentional to prevent merge conflicts, but the team size is over-provisioned for that phase. Observable in retro: workers 2-N report "no unblocked tasks available" during Phase 1 then become active in later phases. Affects Delta interpretation — team efficiency is front-loaded. Mitigation at spec time: (a) consolidate shared-artifact phase into a single task covering all sections (one worker, one pass, no coordination cost), OR (b) ensure the shared-artifact phase is short relative to total work so idling time is bounded. Affects Dimension 1 scoring only if idle workers indicate a knowledge delivery failure (they don't in this case — it's a task design constraint).
